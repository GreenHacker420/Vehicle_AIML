% ============================= ABSTRACT ===========================
\begin{abstract}
Unplanned vehicle breakdowns lead to costly repairs, operational downtime, and safety risks for fleet operators. This paper presents FleetAI, a vehicle maintenance risk prediction system that combines classical machine learning with a rule-based insight engine and optional generative AI enhancement. We train and compare four supervised classifiers on a 50,000-record synthetic vehicle telemetry dataset from Kaggle, selecting a tuned Random Forest pipeline that achieves 94.07\% accuracy and 1.0 ROC-AUC on the full feature set. Each prediction is enriched with structured insight drivers explaining what factors push risk up or down, actionable maintenance recommendations, and data quality warnings. When configured, a Gemini-powered LLM layer refines the natural-language summary and augments the recommendation set. The system is deployed as a FastAPI REST backend supporting JSON and CSV ingestion, with a Next.js dashboard for fleet managers. The complete source code, trained models, and documentation are available at \url{https://github.com/GreenHacker420/Vehicle_AIML}.
\end{abstract}

\begin{IEEEkeywords}
Vehicle maintenance prediction, fleet management, random forest, classification, feature importance, generative AI, Gemini, FastAPI, Next.js, predictive maintenance
\end{IEEEkeywords}

% ============================= INTRODUCTION =======================
\section{Introduction}

Fleet-intensive industries such as logistics, public transit, and ride-hailing depend on vehicle uptime. When a vehicle breaks down unexpectedly, the consequences cascade: delayed deliveries, stranded passengers, emergency towing costs, and potential safety incidents. Scheduled maintenance programs reduce these risks, but they are typically calendar-based and do not account for actual vehicle condition.

Predictive maintenance closes this gap by using operational data to estimate when a vehicle is likely to need servicing. Fleet operators can then prioritize vehicles showing early signs of degradation rather than reacting to failures or following fixed schedules.

However, a raw risk score alone is not enough. Operations teams need to understand \textit{why} a vehicle is flagged and \textit{what} to do about it. Most existing systems stop at the prediction step and leave interpretation to the user.

FleetAI addresses this by combining ML prediction with a structured insight layer. Each prediction includes:
\begin{itemize}
    \item A risk level (\texttt{LOW}, \texttt{MEDIUM}, \texttt{HIGH}) with probability and confidence scores.
    \item Insight drivers explaining which factors increase or decrease risk, with observed values and impact weights.
    \item Actionable maintenance recommendations prioritized by severity.
    \item Data quality warnings when input values fall outside expected ranges.
    \item Optional GenAI-enhanced summaries via Google Gemini for natural-language explanations.
\end{itemize}

Our primary contributions are:
\begin{enumerate}
    \item A comparative evaluation of four classifiers with explicit data-leakage analysis on the Vehicle Maintenance dataset.
    \item A tuned Random Forest pipeline achieving 94.07\% test accuracy with 1.0 ROC-AUC.
    \item A rule-based insight engine that generates structured explanations and recommendations from prediction context.
    \item An optional LLM integration layer using Google Gemini for enhanced natural-language insight generation via LangChain.
    \item A production-ready REST API (FastAPI) and modern dashboard (Next.js) for single and batch predictions, deployed on Vercel.
\end{enumerate}

% ============================= RELATED WORK =======================
\section{Related Work}

Predictive maintenance has been explored across multiple domains. Carvalho et al.~\cite{carvalho2019} provide a broad review of machine-learning approaches for industrial maintenance, highlighting Random Forest and Gradient Boosting for fault classification. Susto et al.~\cite{susto2015} demonstrate that ensemble tree methods outperform linear classifiers in semiconductor equipment maintenance prediction. In the automotive space, Canizo et al.~\cite{canizo2017} apply deep learning to fleet telematics data with strong results but reduced interpretability.

Recent work has explored using large language models (LLMs) for maintenance report generation. Brown et al.~\cite{brown2024} show that combining structured ML outputs with LLM-generated summaries improves operator comprehension and action rates compared to dashboards showing only numerical risk scores.

Our work differs in three ways. First, we investigate data leakage in synthetic datasets and report results with and without leaking features. Second, we build a deterministic rule-based insight engine that works without any external API dependency, ensuring the system remains functional even when GenAI services are unavailable. Third, we layer optional LLM enhancement on top of the rule-based baseline, letting operators benefit from natural-language insights when configured.

\section{Methodology}

\subsection{Dataset Description}

We use the Vehicle Maintenance Data dataset from Kaggle~\cite{kaggle_dataset}, containing 50,000 records with 19 input features and one binary target (\texttt{Need\_Maintenance}). The features span:
\begin{itemize}
    \item \textbf{Vehicle specifications}: model, age, engine size, fuel type, transmission type
    \item \textbf{Operational metrics}: mileage, odometer reading, fuel efficiency
    \item \textbf{Service records}: maintenance history, service history score, days since service, days until warranty expiry
    \item \textbf{Incident data}: reported issues, accident history
    \item \textbf{Component condition}: tire condition, brake condition, battery status
\end{itemize}

The target is imbalanced: approximately 81\% of records are labeled as needing maintenance and 19\% do not.

\subsection{Preprocessing}

Numerical features (11 total) are standardized using \texttt{StandardScaler}. Categorical features (8 total) are transformed via \texttt{OneHotEncoder} with unknown-category handling. Both transformations are composed in a \texttt{ColumnTransformer} and serialized as part of a single scikit-learn \texttt{Pipeline}.

No missing values were present. Date fields (\texttt{Last\_Service\_Date}, \texttt{Warranty\_Expiry}) were converted to integer day counts relative to a fixed reference date before training.

\subsection{Feature Engineering}

Correlation analysis (Figure~\ref{fig:correlation}) revealed \texttt{Reported\_Issues} as the strongest linear predictor (correlation 0.39 with target). \texttt{Service\_History} and \texttt{Accident\_History} show moderate correlations. Most other numerical features exhibit weak linear correlation, indicating the target depends on non-linear feature interactions---a property well-suited for tree-based models.

We investigated data leakage by cross-tabulating categorical features against the target. Component-condition features showed near-deterministic relationships: 100\% of vehicles with ``Worn Out'' brakes or ``Weak'' batteries require maintenance. This confirms these features describe the vehicle's current state from which the target was derived.

We train models on both the full feature set (19 features) and a reduced set (15 features) excluding the four leaking condition features, documenting the performance gap transparently.

\subsection{Models Used}

We evaluate four classifiers:
\begin{enumerate}
    \item \textbf{Logistic Regression}: Linear baseline with L2 regularization and balanced class weights. Serves as a simple, interpretable reference.
    \item \textbf{Decision Tree}: Single tree with balanced class weights. Provides a non-linear baseline with direct interpretability.
    \item \textbf{Random Forest}: Ensemble of 100 bagged decision trees with balanced class weights. Reduces overfitting through averaging.
    \item \textbf{Gradient Boosting}: Sequential boosting with 100 estimators. Builds trees iteratively to correct residual errors.
\end{enumerate}

All models use \texttt{class\_weight='balanced'} to handle the 81/19 class imbalance by automatically adjusting sample weights inversely proportional to class frequencies.

\subsection{Training and Validation}

The dataset is split 80/20 into training (40,000) and test (10,000) sets using stratified sampling to preserve class proportions. Model selection uses 5-fold stratified cross-validation with ROC-AUC as the primary metric. Hyperparameter tuning of the selected model uses \texttt{GridSearchCV} over tree depth, minimum samples per split, minimum samples per leaf, and number of estimators.

\subsection{Backend (FastAPI)}

The backend is built with FastAPI~\cite{fastapi} and follows a layered architecture organized into routes, services, schemas, and models.

\subsubsection{Application Initialization (\texttt{main.py})}
The FastAPI application instance is created with an asynchronous lifespan handler that:
\begin{itemize}
    \item Configures structured logging at startup.
    \item Pre-loads the ML model into memory via \texttt{get\_prediction\_service()}, a singleton cached with \texttt{@lru\_cache(maxsize=1)}.
    \item Registers CORS middleware allowing configurable origins (defaulting to \texttt{["*"]}).
    \item Mounts two route modules: \texttt{predict\_router} and \texttt{vehicles\_router}.
    \item Installs custom exception handlers for \texttt{ValueError} and generic exceptions.
\end{itemize}

\subsubsection{Prediction Endpoint (\texttt{api/predict.py})}
The \texttt{POST /predict} endpoint supports two input modes:
\begin{enumerate}
    \item \textbf{JSON body}: A single vehicle object or an array of objects. Supports wrapper keys \texttt{vehicle\_data} or \texttt{vehicles} for flexible ingestion.
    \item \textbf{CSV file upload}: A multipart file upload with \texttt{field=file}. The CSV is parsed into rows and processed as a batch.
\end{enumerate}

Single requests return a \texttt{PredictionResponse} with one set of results. Batch requests include a \texttt{predictions[]} array with per-row items and a top-level summary representing the highest-risk row.

\subsubsection{Prediction Service (\texttt{services/prediction\_service.py})}
This is the core module (1,021 lines) responsible for all ML inference and insight generation. Key responsibilities:

\begin{enumerate}
    \item \textbf{Model loading}: Loads a serialized \texttt{sklearn.Pipeline} from a \texttt{.pkl} file using \texttt{joblib}. The pipeline encapsulates both preprocessing (StandardScaler~+~OneHotEncoder via ColumnTransformer) and the trained Random Forest classifier.

    \item \textbf{Input normalization}: Accepts flexible input formats (5 minimum fields: \texttt{mileage}, \texttt{engine\_hours}, \texttt{fault\_codes}, \texttt{service\_history}, \texttt{usage\_patterns}) and synthesizes all 19 model features through derivation heuristics:
    \begin{itemize}
        \item Fault code strings are parsed and counted (\texttt{\_extract\_fault\_code\_count}).
        \item Service history text (``good'', ``average'', ``poor'') is mapped to numeric scores (\texttt{\_derive\_service\_history\_score}).
        \item Usage patterns are classified into bands: heavy, moderate, light (\texttt{\_classify\_usage}).
        \item Missing fields (vehicle age, fuel efficiency, insurance premium, component conditions) are derived from available data using domain heuristics.
    \end{itemize}

    \item \textbf{Risk classification}: Model probability is mapped to risk levels using fixed thresholds:
    \begin{itemize}
        \item \texttt{LOW}: probability $< 0.30$
        \item \texttt{MEDIUM}: $0.30 \leq$ probability $< 0.70$
        \item \texttt{HIGH}: probability $\geq 0.70$
    \end{itemize}
    Confidence is computed as $\max(p, 1-p)$ where $p$ is the HIGH-risk class probability.

    \item \textbf{Feature importance extraction}: Extracts model-level (global) feature importance by:
    \begin{itemize}
        \item Reading \texttt{feature\_importances\_} from the Random Forest model.
        \item Mapping one-hot-encoded feature names back to their base categorical feature names.
        \item Aggregating and normalizing importance across the top 10 features.
    \end{itemize}
\end{enumerate}

\subsubsection{Request/Response Schemas (\texttt{schemas/prediction.py})}
All API contracts are defined using Pydantic \texttt{BaseModel} classes with strict typing:

\begin{itemize}
    \item \texttt{VehicleInput}: Accepts 21 optional fields with validation constraints (\texttt{ge=0} for numeric fields). Supports both minimal (5-field) and full (19-field) inputs.
    \item \texttt{PredictionItem}: Core result with \texttt{risk\_level} (Literal type), \texttt{risk\_probability}, \texttt{confidence} (both constrained to [0,1]), \texttt{feature\_importance} dictionary, \texttt{insight\_summary}, typed \texttt{insight\_drivers} list, \texttt{recommendations} list, and \texttt{data\_warnings}.
    \item \texttt{InsightDriver}: Structured driver with \texttt{factor}, \texttt{observed\_value}, \texttt{direction} (RISK\_UP | RISK\_DOWN | NEUTRAL), \texttt{impact} (0--1), and \texttt{explanation}.
    \item \texttt{RecommendationItem}: Action item with \texttt{priority} (HIGH | MEDIUM | LOW), \texttt{action}, and \texttt{rationale}.
    \item \texttt{PredictionResponse}: Extends \texttt{PredictionItem} with \texttt{total\_records}, optional \texttt{predictions[]} array (for batch), and \texttt{metadata} dictionary.
\end{itemize}

\subsection{Insight Engine}

The insight engine is a deterministic, rule-based system within \texttt{PredictionService} that generates row-specific explanations for each prediction. Unlike \textit{global} feature importance (model-wide), insight drivers provide \textit{local} operational interpretation.

\subsubsection{Driver Generation}
For each prediction, nine drivers are evaluated across distinct input dimensions:
\begin{enumerate}
    \item \textbf{Reported issues}: Evaluates active fault code count against thresholds (0, 1, 2, $\geq$5).
    \item \textbf{Service history}: Maps service score (0--10 scale) to risk direction.
    \item \textbf{Service recency}: Evaluates days since last service ($<$120, 120--240, $\geq$240 days).
    \item \textbf{Brake condition}: Maps component state (New/Good/Worn Out) to risk impact.
    \item \textbf{Tire condition}: Same mapping as brake condition.
    \item \textbf{Battery status}: Maps battery health (New/Good/Weak) to risk.
    \item \textbf{Accident history}: Evaluates number of past incidents (0, 1, $\geq$2).
    \item \textbf{Mileage}: Categorizes into bands ($<$120k, 120k--250k, $\geq$250k km).
    \item \textbf{Usage profile}: Distinguishes commercial vs. individual operation.
\end{enumerate}

Each driver produces an \texttt{InsightDriver} object with a calibrated impact score (0.0--1.0) and a direction label. Drivers are sorted by impact score in descending order, and the top 6 are included in the response.

\subsubsection{Recommendation Generation}
Recommendations are generated by mapping active RISK\_UP drivers to specific maintenance actions:
\begin{itemize}
    \item Fault codes $\rightarrow$ ``Run full diagnostic scan''
    \item Extended service interval $\rightarrow$ ``Schedule preventive service within 7 days''
    \item Degraded components $\rightarrow$ ``Inspect and service [component] immediately''
    \item Poor service history $\rightarrow$ ``Move to shorter maintenance intervals''
    \item High mileage $\rightarrow$ ``Increase preventive inspection frequency''
\end{itemize}

Recommendations are deduplicated and capped at 5 items. If no drivers trigger a recommendation, a default low-priority item is returned.

\subsubsection{Data Quality Warnings}
Input values are checked against expected ranges, generating warnings for:
\begin{itemize}
    \item Mileage $> 1{,}500{,}000$ km (outside training range)
    \item Mileage-to-engine-hours ratio $> 2500$ (telemetry inconsistency)
    \item Reported issues $\geq 15$ (immediate workshop inspection recommended)
\end{itemize}

\subsection{GenAI Integration (\texttt{llm\_insight\_service.py})}

An optional LLM layer enhances predictions when configured with a Google API key. The \texttt{LLMInsightService} class:

\begin{enumerate}
    \item \textbf{Activation}: Enabled only when \texttt{ENABLE\_LLM\_INSIGHTS=true} and a valid \texttt{GOOGLE\_API\_KEY} is set. If not configured, the service is silently disabled.
    \item \textbf{LLM Initialization}: Uses LangChain's \texttt{ChatGoogleGenerativeAI} wrapper with the Gemini model (default: \texttt{gemini-1.5-flash}) at temperature 0.2 for consistent output.
    \item \textbf{Prompt Design}: Sends a system prompt positioning the model as a ``senior fleet maintenance analyst'' and a human prompt containing the full prediction context (risk level, probability, rule summary, normalized row, and drivers) as JSON.
    \item \textbf{Response Parsing}: Extracts JSON from the LLM response, handling markdown code fences and partial JSON. Returns a \texttt{summary} string and up to 3 \texttt{recommendations}.
    \item \textbf{Recommendation Merging}: LLM recommendations are prepended to rule-based recommendations, deduplicated by action text, and capped at 5 total items.
    \item \textbf{Fallback}: If the LLM call fails, times out, or returns unparseable output, the system gracefully falls back to rule-based insights with zero degradation.
    \item \textbf{Source Tracking}: The \texttt{insight\_source} field switches from \texttt{RULES} to \texttt{GENAI\_LLM} so the frontend can visually indicate the source.
\end{enumerate}

\subsection{Frontend (Next.js)}

The frontend is built with Next.js~\cite{nextjs} using TypeScript, TailwindCSS, and the shadcn/ui component library. It provides a modern, responsive dashboard for fleet managers.

\subsubsection{Landing Page (\texttt{page.tsx})}
A full-screen hero page with animated gradient backgrounds (WebGL shader), text generation effects, and a call-to-action button linking to the prediction workspace. Uses custom components:
\begin{itemize}
    \item \texttt{ShaderBackground}: WebGL-powered animated gradient background.
    \item \texttt{TextGenerateEffect}: Character-by-character text reveal animation.
    \item \texttt{HoverBorderGradient}: Animated border gradient on interactive buttons.
    \item \texttt{AnimatedContent} / \texttt{FadeContent}: Scroll-triggered entrance animations.
\end{itemize}

\subsubsection{Prediction Page (\texttt{predict/page.tsx})}
The 727-line prediction workspace is the core interface, providing:

\begin{enumerate}
    \item \textbf{Manual Input Form}: Five fields (mileage, engine hours, fault codes, service history, usage patterns) with client-side validation, styled with custom Input and Textarea components.

    \item \textbf{CSV Upload}: Drag-and-drop file upload via \texttt{FileUpload} component with file type validation (\texttt{.csv} only).

    \item \textbf{State Management}: Uses TanStack Query (React Query) \texttt{useMutation} for API calls with loading states, error handling, and success callbacks.

    \item \textbf{Results Dashboard}: Upon successful prediction, displays:
    \begin{itemize}
        \item Risk level badge (color-coded: emerald/amber/rose)
        \item Insight source indicator (Rule vs. GenAI)
        \item Insight summary text
        \item Risk probability and confidence metrics
        \item Total records count
        \item Data quality warnings (if any)
        \item Batch preview table (for CSV uploads, limited to 8 rows)
        \item Meaningful Insight table: drivers with factor name, observed value, direction badge (with arrow icons), and impact percentage
        \item Recommended Actions cards with priority badges and rationale
        \item Feature Importance table showing global model weights
    \end{itemize}

    \item \textbf{Informational Accordion}: Collapsible sections explaining input schema, performance characteristics, risk thresholds, and the distinction between global feature importance vs. row-specific insight drivers.
\end{enumerate}

\subsubsection{Service Layer (\texttt{predictionService.ts})}
TypeScript service module providing two async functions:
\begin{itemize}
    \item \texttt{predictFromManualInput(payload)}: Sends JSON POST to \texttt{/predict}.
    \item \texttt{predictFromCsvFile(file)}: Sends multipart POST with CSV file to \texttt{/predict}.
\end{itemize}

The API base URL is resolved from the \texttt{NEXT\_PUBLIC\_API\_BASE\_URL} environment variable, with automatic fallback to \texttt{localhost:8000} during development and the deployed Vercel backend URL in production.

\subsubsection{Type Definitions (\texttt{types/prediction.ts})}
Full TypeScript types mirror the backend Pydantic schemas, ensuring type-safe communication:
\begin{itemize}
    \item \texttt{ManualPredictionInput}: Input fields with correct types.
    \item \texttt{PredictionItem}: Core result with literal union types for risk levels and directions.
    \item \texttt{InsightDriver}: Driver structure matching the backend \texttt{InsightDriver} schema.
    \item \texttt{RecommendationItem}: Action items with priority typing.
    \item \texttt{PredictionResponse}: Extends PredictionItem with batch fields.
\end{itemize}

\subsection{ML Pipeline}

The trained model is serialized as a single scikit-learn \texttt{Pipeline} object (\texttt{vehicle\_maintenance\_pipeline.pkl}) that encapsulates:

\begin{enumerate}
    \item \textbf{Preprocessor} (\texttt{ColumnTransformer}):
    \begin{itemize}
        \item \texttt{num}: \texttt{StandardScaler} applied to 11 numerical features.
        \item \texttt{cat}: \texttt{OneHotEncoder} with \texttt{handle\_unknown='ignore'} applied to 8 categorical features.
    \end{itemize}
    \item \textbf{Model}: Tuned \texttt{RandomForestClassifier} with \texttt{max\_depth=8}, \texttt{min\_samples\_leaf=10}, \texttt{min\_samples\_split=5}, \texttt{n\_estimators=100}, and \texttt{class\_weight='balanced'}.
\end{enumerate}

The notebook \texttt{model/data\_processing.ipynb} contains the complete EDA, model training, evaluation, and plot generation workflow.

% ============================= DISCUSSION =========================
\section{Discussion}

\subsection{Model Selection}

We selected Random Forest over Decision Tree and Gradient Boosting for deployment despite Decision Tree and Gradient Boosting achieving perfect accuracy on the full feature set. Random Forest provides more robust feature importance scores, generalizes better to unseen distributions, and is less prone to overfitting on a synthetic dataset where perfect accuracy is itself an artifact of data construction. The tuned hyperparameters (\texttt{max\_depth=8}, \texttt{min\_samples\_leaf=10}) further constrain tree complexity to improve generalization.

\subsection{Data Leakage Analysis}

The 30+ percentage point accuracy gap between the full and realistic feature sets (Tables~\ref{tab:full} and~\ref{tab:realistic}) demonstrates the critical importance of investigating feature-target relationships before reporting results. Component-condition features (\texttt{Brake\_Condition}, \texttt{Battery\_Status}, \texttt{Tire\_Condition}) are effectively proxies for the target variable---knowing that brakes are ``Worn Out'' already implies that maintenance is needed.

In a real deployment, these features might not be available at prediction time since knowing the exact brake condition already answers whether maintenance is needed. We retain the full feature set in the deployed pipeline because the project specification requires using the complete dataset, but document this limitation clearly so users understand the performance context.

\subsection{Insight Engine Design}

Raw risk scores lack actionable context. A fleet manager seeing ``HIGH risk, 87\% probability'' cannot determine next steps without understanding the underlying cause. Our insight engine addresses this by:
\begin{enumerate}
    \item Evaluating each input dimension against threshold-based rules to produce typed \texttt{InsightDriver} objects with direction (RISK\_UP, RISK\_DOWN, NEUTRAL) and calibrated impact weights.
    \item Ranking drivers by impact and selecting the top six for inclusion in the response.
    \item Generating prioritized \texttt{RecommendationItem} objects (HIGH, MEDIUM, LOW) based on which drivers are active.
    \item Flagging data quality warnings when inputs fall outside expected ranges.
\end{enumerate}

This rule-based layer runs deterministically with zero external dependencies, ensuring consistent behavior across all deployment environments. The separation between \textit{global feature importance} (model-wide) and \textit{row-specific insight drivers} (operational) is a key design decision that prevents confusion when users see high model importance for a feature that has a neutral value for their specific vehicle.

\subsection{GenAI Enhancement}

The optional Gemini integration demonstrates a practical GenAI integration pattern:
\begin{itemize}
    \item The rule-based baseline provides a reliable, always-available foundation.
    \item The LLM enhances but never replaces the core output.
    \item Graceful degradation ensures the system works identically with or without LLM configuration.
    \item The \texttt{insight\_source} field provides transparency about which system generated the insights.
\end{itemize}

The prompt is specifically designed to produce structured JSON output, and the parsing logic handles common LLM output quirks (markdown fences, partial JSON) robustly.

\subsection{Limitations}

\begin{itemize}
    \item The dataset is synthetic. Real fleet data would contain noise, missing values, and distribution shifts not present here.
    \item Class imbalance is handled via balanced class weights. Techniques like SMOTE or cost-sensitive learning could be explored for potentially better minority-class recall.
    \item The insight engine uses fixed thresholds tuned to this dataset. Adapting to different fleet profiles would require threshold calibration.
    \item LLM-generated recommendations are not validated against domain-specific maintenance standards (e.g., OEM service manuals).
    \item The deployed model includes leaking features. In a production scenario, component-condition features should be removed or replaced with sensor-derived equivalents available before maintenance diagnosis.
\end{itemize}

\section{Deployment}

FleetAI is deployed on Vercel using a two-project monorepo strategy:

\subsection{Backend Deployment}
\begin{itemize}
    \item \textbf{Entrypoint}: \texttt{api/index.py} serves as the Vercel serverless function entrypoint. It configures \texttt{sys.path} to locate the \texttt{backend/} module and imports the FastAPI \texttt{app} instance.
    \item \textbf{Routing}: \texttt{vercel.json} at the repository root routes all requests to the Python serverless function.
    \item \textbf{Dependencies}: \texttt{requirements.txt} at the root specifies minimal dependencies: FastAPI, pandas, joblib, scikit-learn, python-multipart, pydantic, pydantic-settings, and numpy.
    \item \textbf{Model artifact}: The \texttt{model/vehicle\_maintenance\_pipeline.pkl} file (serialized sklearn pipeline) is bundled with the deployment.
    \item \textbf{Optional GenAI}: Set \texttt{ENABLE\_LLM\_INSIGHTS=true}, \texttt{GOOGLE\_API\_KEY}, and \texttt{GENAI\_MODEL} as Vercel environment variables to enable LLM insights.
\end{itemize}

\subsection{Frontend Deployment}
\begin{itemize}
    \item \textbf{Vercel project root}: \texttt{frontend/} directory.
    \item \textbf{Environment variable}: \texttt{NEXT\_PUBLIC\_API\_BASE\_URL} points to the deployed backend domain.

\end{itemize}

\end{itemize}

% ============================= CONCLUSION =========================
\section{Conclusion}

We presented FleetAI, a vehicle maintenance prediction system that combines ML classification with structured insight generation and optional GenAI enhancement. Our analysis of the Kaggle Vehicle Maintenance dataset uncovered significant data leakage from component-condition features, and we reported results on both the full and realistic feature sets for transparency. A tuned Random Forest classifier achieves 94.07\% accuracy and is deployed as a REST API supporting single JSON and batch CSV predictions. Each prediction is enriched with insight drivers, actionable recommendations, and data quality warnings through a deterministic rule engine. An optional Gemini LLM layer enhances natural-language summaries when configured. The system is production-deployed on Vercel with a modern Next.js dashboard featuring animated UI components and responsive design.

Future work includes:
\begin{itemize}
    \item RAG-based retrieval of maintenance manuals via Pinecone for contextually grounded recommendations.
    \item Agentic workflow orchestration using LangGraph for multi-step analysis.
    \item Persistent vehicle health tracking with PostgreSQL via Piccolo ORM (table definitions already exist in the codebase).
    \item Model retraining with real-world fleet data and SMOTE for improved class imbalance handling.
    \item Mobile application for field technicians.
\end{itemize}
