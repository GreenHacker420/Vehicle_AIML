% ---------- ABSTRACT ----------
\begin{abstract}
Unplanned vehicle breakdowns lead to costly repairs, operational downtime, and safety risks for fleet operators. This paper presents FleetAI, a vehicle maintenance risk prediction system that combines classical machine learning with a rule-based insight engine and optional generative AI enhancement. We train and compare four supervised classifiers on a 50,000-record synthetic vehicle telemetry dataset from Kaggle, selecting a tuned Random Forest pipeline that achieves 94.07\% accuracy and 1.0 ROC-AUC on the full feature set. Each prediction is enriched with structured insight drivers explaining what factors push risk up or down, actionable maintenance recommendations, and data quality warnings. When configured, a Gemini-powered LLM layer refines the natural-language summary and augments the recommendation set. The system is deployed as a FastAPI REST backend supporting JSON and CSV ingestion, with a Next.js dashboard for fleet managers.
\end{abstract}

\begin{IEEEkeywords}
Vehicle maintenance prediction, fleet management, random forest, classification, feature importance, generative AI, Gemini, FastAPI
\end{IEEEkeywords}

% ---------- INTRODUCTION ----------
\section{Introduction}

Fleet-intensive industries such as logistics, public transit, and ride-hailing depend on vehicle uptime. When a vehicle breaks down unexpectedly, the consequences cascade: delayed deliveries, stranded passengers, emergency towing costs, and potential safety incidents. Scheduled maintenance programs reduce these risks, but they are typically calendar-based and do not account for actual vehicle condition.

Predictive maintenance closes this gap by using operational data to estimate when a vehicle is likely to need servicing. Fleet operators can then prioritize vehicles showing early signs of degradation rather than reacting to failures or following fixed schedules.

However, a raw risk score alone is not enough. Operations teams need to understand \textit{why} a vehicle is flagged and \textit{what} to do about it. Most existing systems stop at the prediction step and leave interpretation to the user.

FleetAI addresses this by combining ML prediction with a structured insight layer. Each prediction includes:
\begin{itemize}
    \item A risk level (Low, Medium, High) with probability and confidence scores.
    \item Insight drivers explaining which factors increase or decrease risk, with observed values and impact weights.
    \item Actionable maintenance recommendations prioritized by severity.
    \item Data quality warnings when input values fall outside expected ranges.
    \item Optional GenAI-enhanced summaries via Google Gemini for natural-language explanations.
\end{itemize}

Our primary contributions are:
\begin{enumerate}
    \item A comparative evaluation of four classifiers with explicit data-leakage analysis on the Vehicle Maintenance dataset.
    \item A tuned Random Forest pipeline achieving 94.07\% test accuracy.
    \item A rule-based insight engine that generates structured explanations and recommendations from prediction context.
    \item An optional LLM integration layer using Gemini for enhanced natural-language insight generation.
    \item A production-ready REST API and dashboard for single and batch predictions.
\end{enumerate}

% ---------- RELATED WORK ----------
\section{Related Work}

Predictive maintenance has been explored across multiple domains. Carvalho et al.~\cite{carvalho2019} provide a broad review of machine-learning approaches for industrial maintenance, highlighting Random Forest and Gradient Boosting for fault classification. Susto et al.~\cite{susto2015} demonstrate that ensemble tree methods outperform linear classifiers in semiconductor equipment maintenance prediction. In the automotive space, Canizo et al.~\cite{canizo2017} apply deep learning to fleet telematics data with strong results but reduced interpretability.

Recent work has explored using large language models (LLMs) for maintenance report generation. Brown et al.~\cite{brown2024} show that combining structured ML outputs with LLM-generated summaries improves operator comprehension and action rates compared to dashboards showing only numerical risk scores.

Our work differs in three ways. First, we investigate data leakage in synthetic datasets and report results with and without leaking features. Second, we build a deterministic rule-based insight engine that works without any external API dependency, ensuring the system remains functional even when GenAI services are unavailable. Third, we layer optional LLM enhancement on top of the rule-based baseline, letting operators benefit from natural-language insights when configured.

% ---------- METHODOLOGY ----------
\section{Methodology}

\subsection{Dataset Description}

We use the Vehicle Maintenance Data dataset from Kaggle~\cite{kaggle_dataset}, containing 50,000 records with 19 input features and one binary target (\texttt{Need\_Maintenance}). The features span:
\begin{itemize}
    \item \textbf{Vehicle specs}: model, age, engine size, fuel type, transmission type
    \item \textbf{Operational metrics}: mileage, odometer reading, fuel efficiency
    \item \textbf{Service records}: maintenance history, service history score, days since service, days until warranty
    \item \textbf{Incident data}: reported issues, accident history
    \item \textbf{Component condition}: tire condition, brake condition, battery status
\end{itemize}

The target is imbalanced: approximately 81\% of records are labeled as needing maintenance and 19\% do not.

\subsection{Preprocessing}

Numerical features (11 total) are standardized using \texttt{StandardScaler}. Categorical features (8 total) are transformed via \texttt{OneHotEncoder} with unknown-category handling. Both transformations are composed in a \texttt{ColumnTransformer} and serialized as part of a single sklearn \texttt{Pipeline}.

No missing values were present. Date fields (\texttt{Last\_Service\_Date}, \texttt{Warranty\_Expiry}) were converted to integer day counts relative to a fixed reference date before training.

\subsection{Feature Engineering}

Correlation analysis revealed \texttt{Reported\_Issues} as the strongest linear predictor (correlation 0.39 with target). \texttt{Service\_History} and \texttt{Accident\_History} show moderate correlations. Most other numerical features exhibit weak linear correlation, indicating the target depends on non-linear interactions.

We investigated data leakage by cross-tabulating categorical features against the target. Component-condition features showed near-deterministic relationships: 100\% of vehicles with ``Worn Out'' brakes or ``Weak'' batteries require maintenance. This confirms these features describe the vehicle's current state from which the target was derived.

We train models on both the full feature set (19 features) and a reduced set (15 features) excluding the four leaking condition features, documenting the performance gap.

\subsection{Models Used}

We evaluate four classifiers:
\begin{enumerate}
    \item \textbf{Logistic Regression}: Linear baseline with L2 regularization and balanced class weights.
    \item \textbf{Decision Tree}: Single tree with balanced class weights.
    \item \textbf{Random Forest}: Ensemble of 100 trees with balanced class weights.
    \item \textbf{Gradient Boosting}: Sequential boosting with 100 estimators.
\end{enumerate}

All models use \texttt{class\_weight='balanced'} to handle the 81/19 imbalance.

\subsection{Training and Validation}

The dataset is split 80/20 into training (40,000) and test (10,000) sets using stratified sampling. Model selection uses 5-fold stratified cross-validation with ROC-AUC as the primary metric. Hyperparameter tuning of the selected model uses \texttt{GridSearchCV} over tree depth, minimum samples per split, minimum samples per leaf, and number of estimators.

% ---------- DISCUSSION ----------
\section{Discussion}

\subsection{Model Selection}

We selected Random Forest over Decision Tree and Gradient Boosting for deployment despite Decision Tree and Gradient Boosting achieving perfect accuracy on the full feature set. Random Forest provides more robust feature importance scores, generalizes better to unseen distributions, and is less prone to overfitting on a synthetic dataset where perfect accuracy is itself an artifact of data construction.

\subsection{Data Leakage}

The 30+ percentage point accuracy gap between the full and realistic feature sets demonstrates the importance of investigating feature-target relationships before reporting results. Component-condition features (\texttt{Brake\_Condition}, \texttt{Battery\_Status}, \texttt{Tire\_Condition}) are effectively proxies for the target variable. In a real deployment, these features might not be available at prediction time since knowing the exact brake condition already answers whether maintenance is needed.

We retain the full feature set in the deployed pipeline because the project specification requires using the complete dataset, but document this limitation clearly.

\subsection{Insight Engine Design}

Raw risk scores lack actionable context. A fleet manager seeing ``HIGH risk, 87\% probability'' cannot determine next steps without understanding the underlying cause. Our insight engine addresses this by:
\begin{enumerate}
    \item Evaluating each input dimension against threshold-based rules to produce typed \texttt{InsightDriver} objects with direction (RISK\_UP, RISK\_DOWN, NEUTRAL) and impact weights.
    \item Ranking drivers by impact and selecting the top six for inclusion in the response.
    \item Generating prioritized \texttt{RecommendationItem} objects (HIGH, MEDIUM, LOW) based on which drivers are active.
    \item Flagging data quality warnings when inputs fall outside expected ranges.
\end{enumerate}

This rule-based layer runs deterministically with zero external dependencies, ensuring consistent behavior across all deployment environments.

\subsection{GenAI Integration}

When configured with a Google API key, an optional LLM layer enhances predictions:
\begin{itemize}
    \item The rule-based insight summary, input row, and driver details are sent to Gemini as structured context.
    \item Gemini returns a refined natural-language summary and up to three additional recommendations.
    \item LLM recommendations are merged with rule-based recommendations, deduplicated, and capped at five items.
    \item The \texttt{insight\_source} field switches from \texttt{RULES} to \texttt{GENAI\_LLM} so the frontend can indicate the source.
\end{itemize}

If the LLM call fails or is not configured, the response falls back to rule-based insights with no degradation in functionality.

\subsection{Limitations}

\begin{itemize}
    \item The dataset is synthetic. Real fleet data would contain noise, missing values, and distribution shifts not present here.
    \item Class imbalance is handled via balanced class weights. Techniques like SMOTE or cost-sensitive learning could be explored.
    \item The insight engine uses fixed thresholds tuned to this dataset. Adapting to different fleet profiles would require threshold calibration.
    \item LLM-generated recommendations are not validated against domain-specific maintenance standards.
\end{itemize}

% ---------- CONCLUSION ----------
\section{Conclusion}

We presented FleetAI, a vehicle maintenance prediction system combining ML classification with structured insight generation. Our analysis of the Kaggle Vehicle Maintenance dataset uncovered data leakage from component-condition features, and we reported results on both the full and realistic feature sets for transparency. A tuned Random Forest classifier achieves 94.07\% accuracy and is deployed as a REST API supporting single JSON and batch CSV predictions. Each prediction is enriched with insight drivers, recommendations, and data quality warnings through a deterministic rule engine. An optional Gemini LLM layer enhances natural-language summaries when configured. Future work includes RAG-based retrieval of maintenance manuals via Pinecone, agentic workflow orchestration using LangGraph, and persistent vehicle health tracking with PostgreSQL.
