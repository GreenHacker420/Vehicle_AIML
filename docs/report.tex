% ============================= ABSTRACT ===========================
\begin{abstract}
Unplanned vehicle breakdowns cost fleet operators thousands of dollars in repairs, idle time, and missed deliveries every year. In this paper we present FleetAI, a system that predicts whether a vehicle is likely to need maintenance by combining classical machine learning with a structured insight engine and an optional generative AI layer. We trained and compared four supervised classifiers on a publicly available Kaggle dataset of 50,000 synthetic vehicle records. A tuned Random Forest pipeline was selected for deployment, achieving 94.07\% accuracy and a perfect 1.0 ROC AUC on the full feature set. Beyond the raw prediction, every result includes human readable insight drivers that explain which factors are pushing risk up or down, a prioritized list of recommended maintenance actions, and warnings when the input data looks unusual. When a Google Gemini API key is configured, the system also produces a polished natural language summary of the prediction. The whole stack is deployed as a FastAPI backend and a Next.js dashboard, both hosted on Vercel. All source code, the trained model, and project documentation are openly available at \url{https://github.com/GreenHacker420/Vehicle_AIML}.
\end{abstract}

\begin{IEEEkeywords}
Vehicle maintenance prediction, fleet management, random forest, classification, feature importance, generative AI, Gemini, FastAPI, Next.js, predictive maintenance
\end{IEEEkeywords}

% ============================= INTRODUCTION =======================
\section{Introduction}

Anyone who has managed a fleet of vehicles knows how expensive surprise breakdowns can be. A delivery truck that stalls on a highway does not just need a tow; it delays every package on board, strains the rest of the fleet, and can put the driver at risk. Most companies try to avoid this by scheduling maintenance at fixed intervals, but that approach treats every vehicle the same regardless of how it is actually being used.

Predictive maintenance offers a smarter alternative. Instead of servicing vehicles on a calendar, the idea is to use operational data to estimate which vehicles are most likely to fail soon and prioritize them accordingly. This lets fleet managers allocate their maintenance budgets where they matter most.

The problem is that a bare risk score on its own is not very helpful. If the system says ``this truck has an 87\% chance of needing maintenance,'' the operations team still needs to know \textit{why} and \textit{what to do about it}. Most academic prototypes stop at the prediction and leave interpretation entirely to the user.

We built FleetAI to fill that gap. Every prediction that comes out of our system includes three things beyond the risk score itself. First, a set of structured insight drivers that tell the user exactly which input factors contributed to the risk and in which direction. Second, a list of concrete maintenance recommendations ranked by urgency. Third, data quality warnings that flag anything unusual about the input, such as mileage values that fall outside the range our model was trained on. On top of all that, when a Google Gemini API key is provided, the system produces a polished summary written in natural language by the LLM.

The main contributions of this work are as follows:

\begin{enumerate}
    \item We compare four classifiers on the Vehicle Maintenance dataset and explicitly investigate data leakage, reporting results on both the full and clean feature sets.
    \item We train and tune a Random Forest pipeline that reaches 94.07\% accuracy on the test set.
    \item We design a deterministic, rule based insight engine that generates structured explanations and recommendations with zero external dependencies.
    \item We integrate an optional LLM layer powered by Google Gemini through LangChain, with graceful fallback to the rule based engine if the LLM is unavailable.
    \item We deploy the complete system as a production ready REST API with a modern web dashboard, both running on Vercel.
\end{enumerate}

% ============================= RELATED WORK =======================
\section{Related Work}

The idea of using machine learning for maintenance prediction has been explored extensively across several industries. Carvalho et al.~\cite{carvalho2019} surveyed the literature and found that ensemble methods such as Random Forest and Gradient Boosting consistently perform well for fault classification tasks. Susto et al.~\cite{susto2015} reached a similar conclusion when predicting semiconductor equipment failures, showing that tree ensembles outperform linear models by a significant margin. In the automotive domain specifically, Canizo et al.~\cite{canizo2017} applied deep learning to fleet telematics data and achieved strong predictive performance, although at the cost of model interpretability.

More recently, researchers have started experimenting with large language models to make maintenance outputs easier for non technical users to understand. Brown et al.~\cite{brown2024} demonstrated that pairing structured ML predictions with LLM generated summaries measurably improved operator comprehension and the rate at which they took corrective action, compared to dashboards that only showed numerical scores.

Our work builds on these findings but takes a different approach in three respects. First, we explicitly test for data leakage in the synthetic dataset and report honest numbers for both the leaked and clean scenarios. Second, rather than relying on an LLM as the sole explanation mechanism, we implemented a fully deterministic rule based insight engine that runs without any external API calls. Third, we treat the LLM as an optional enhancement layer that sits on top of the rule engine, so the system degrades gracefully when the Gemini service is not available.

% ============================= METHODOLOGY ========================
\section{Methodology}

\subsection{Dataset Description}

We used the Vehicle Maintenance Data dataset published on Kaggle by Dulaj~\cite{kaggle_dataset}. It contains 50,000 synthetically generated vehicle records with 19 input features and a single binary target column called \texttt{Need\_Maintenance}. The features cover a broad range of vehicle characteristics:

\begin{itemize}
    \item Vehicle specifications such as model name, age, engine size, fuel type, and transmission type.
    \item Operational metrics including mileage, odometer reading, and fuel efficiency.
    \item Service records like maintenance history counts, a service history score, days since the last service visit, and days remaining until warranty expiry.
    \item Incident data covering the number of reported issues and past accident history.
    \item Component condition indicators for tires, brakes, and battery.
\end{itemize}

One important characteristic of this dataset is the class imbalance. Roughly 81\% of the records are labeled as needing maintenance and only about 19\% are not. We took this into account during model training, as described below.

\subsection{Preprocessing}

The 11 numerical features were standardized using \texttt{StandardScaler} so that each one has zero mean and unit variance. The 8 categorical features were converted into binary indicator columns using \texttt{OneHotEncoder} with unknown category handling enabled. We wrapped both transformations inside a \texttt{ColumnTransformer} and chained the result with the classifier into a single scikit learn \texttt{Pipeline}, which was then serialized to disk.

There were no missing values in the dataset. The two date columns (\texttt{Last\_Service\_Date} and \texttt{Warranty\_Expiry}) were converted to integer day counts relative to a fixed reference date so the model could work with them numerically.

\subsection{Feature Engineering}

Looking at the correlation heatmap in Figure~\ref{fig:correlation}, we noticed that \texttt{Reported\_Issues} stood out as the strongest linear predictor, with a correlation of 0.39. \texttt{Service\_History} and \texttt{Accident\_History} showed moderate correlations. The rest of the numerical features had fairly weak linear relationships with the target, which pointed toward nonlinear interactions as the main signal. This is exactly the kind of pattern where tree based models tend to shine.

We also investigated potential data leakage by cross tabulating the categorical features against the target. What we found was concerning: component condition features had an almost deterministic relationship with the label. For example, 100\% of vehicles with ``Worn Out'' brakes and 100\% of vehicles with ``Weak'' batteries were labeled as needing maintenance. This makes sense when you think about it, because knowing the exact brake or battery condition essentially gives away the answer.

To be transparent about this, we trained and evaluated models on two separate feature sets. The full set includes all 19 features, while the realistic set drops the four leaking condition columns and uses only 15 features.

\subsection{Models Used}

We evaluated four classifiers to get a sense of how different modeling approaches handle this particular dataset:

\begin{enumerate}
    \item \textbf{Logistic Regression}: A simple linear model with L2 regularization and balanced class weights. We included this primarily as an interpretable baseline.
    \item \textbf{Decision Tree}: A single tree classifier with balanced class weights. It captures nonlinear patterns but is prone to overfitting.
    \item \textbf{Random Forest}: An ensemble of 100 bagged decision trees with balanced class weights. By averaging many trees, it reduces variance and tends to generalize better.
    \item \textbf{Gradient Boosting}: A sequential ensemble of 100 boosted trees. Each new tree focuses on correcting the mistakes of the previous ones.
\end{enumerate}

Every model was configured with balanced class weights so that the minority class (no maintenance needed) would not be drowned out by the majority.

\subsection{Training and Validation}

We split the 50,000 records into 80\% for training (40,000 records) and 20\% for testing (10,000 records) using stratified sampling to keep the class proportions consistent. Model selection was based on 5 fold stratified cross validation with ROC AUC as the primary metric.

After selecting Random Forest as the best candidate, we fine tuned its hyperparameters using \texttt{GridSearchCV}. The search covered tree depth, minimum samples per split, minimum samples per leaf, and the number of estimators.

\subsection{Backend (FastAPI)}

We chose FastAPI~\cite{fastapi} for the backend because it offers automatic request validation through Pydantic, built in OpenAPI documentation, and excellent performance for a Python framework. The code is organized into four layers: routes, services, schemas, and configuration.

\subsubsection{Application Initialization}
The main application file (\texttt{main.py}) sets up an asynchronous lifespan handler that loads the ML model into memory when the server starts. The model is wrapped in a singleton service cached with Python's \texttt{@lru\_cache} decorator, so it only gets loaded once. The file also configures CORS middleware, registers the prediction and vehicle route modules, and installs custom error handlers that return structured JSON responses for both validation errors and unexpected exceptions.

\subsubsection{Prediction Endpoint}
The \texttt{POST /predict} endpoint is flexible enough to handle two very different input styles. You can send a single vehicle as a JSON object, or you can send an array of vehicles, optionally wrapped inside a \texttt{vehicle\_data} or \texttt{vehicles} key. Alternatively, you can upload a CSV file as a multipart form submission. The endpoint figures out what it received and routes the request to the appropriate handler.

For single requests, it returns one prediction with all the insight information. For batch requests, the response includes a \texttt{predictions[]} array with individual results for every row, plus a top level summary that represents the highest risk vehicle in the batch.

\subsubsection{Prediction Service}
The prediction service file is the heart of the entire backend at 1,021 lines of Python. It handles several responsibilities.

First, it loads the serialized scikit learn pipeline from a pickle file using joblib. The pipeline contains both the preprocessing steps (scaling and encoding) and the trained Random Forest classifier, so a single \texttt{predict\_proba} call handles everything.

Second, it normalizes whatever input it receives into the 19 features the model expects. The API only requires five fields (mileage, engine hours, fault codes, service history, and usage patterns), and the service derives everything else through domain heuristics. For example, it parses fault code strings like ``P0171,P0420'' and counts them, maps service history text (``good'', ``average'', ``poor'') to numeric scores, and classifies usage patterns into heavy, moderate, or light bands.

Third, it maps the model's probability output to one of three risk levels: \texttt{LOW} for probabilities below 0.30, \texttt{MEDIUM} for probabilities between 0.30 and 0.70, and \texttt{HIGH} for anything at or above 0.70.

Finally, it extracts global feature importance from the Random Forest by reading the \texttt{feature\_importances\_} attribute, mapping one hot encoded column names back to their original feature names, aggregating across categories, and returning the top 10 in normalized form.

\subsubsection{Configuration}
Application settings are managed through Pydantic Settings with the \texttt{FLEET\_} environment variable prefix. The model file path is automatically detected by searching three candidate locations, which makes the same code work both in local development and in a Vercel serverless deployment without any changes.

\subsubsection{Request and Response Schemas}
Every field in every API request and response is typed using Pydantic models. The \texttt{VehicleInput} model accepts up to 21 optional fields with numeric validation constraints. The \texttt{PredictionItem} model defines the output structure including the risk level (as a Literal union type), probability and confidence scores (constrained to the 0 to 1 range), feature importance dictionary, insight summary text, typed insight driver objects, recommendation items, and data warnings. These schemas serve as both runtime validation and living documentation.

\subsection{Insight Engine}

We wanted every prediction to come with an explanation, not just a number. The insight engine is a deterministic, rule based system embedded inside the prediction service that generates row specific explanations. This is fundamentally different from global feature importance, which tells you what matters across the entire model. Insight drivers tell you what matters \textit{for this specific vehicle}.

\subsubsection{Driver Generation}
For each prediction, the engine evaluates nine separate input dimensions and produces an \texttt{InsightDriver} object for each one. These dimensions are: reported issues (fault code count), service history (quality score), service recency (days since last visit), brake condition, tire condition, battery status, accident history, mileage band, and usage profile (commercial versus individual). Each driver gets a direction label (\texttt{RISK\_UP}, \texttt{RISK\_DOWN}, or \texttt{NEUTRAL}) and a calibrated impact score between 0.0 and 1.0. The drivers are sorted by impact and the top six are included in the final response.

\subsubsection{Recommendation Generation}
When a driver has a \texttt{RISK\_UP} direction, the engine maps it to a specific maintenance action. Fault codes trigger a ``run full diagnostic scan'' recommendation. An extended service interval triggers ``schedule preventive service within 7 days.'' Degraded components trigger an inspection recommendation for that specific part. These are deduplicated and capped at five items. If nothing triggers, a low priority default recommendation is returned instead.

\subsubsection{Data Quality Warnings}
The engine also checks whether input values fall within reasonable ranges and flags anything suspicious. For instance, mileage above 1,500,000 km gets flagged because it exceeds the training data range. A mileage to engine hours ratio above 2,500 raises a telemetry inconsistency warning. And if a vehicle reports 15 or more active issues, the system warns that immediate workshop inspection may be warranted.

\subsection{GenAI Integration}

On top of the rule based insight engine, we built an optional LLM layer that can enhance the prediction output with a natural language summary. The \texttt{LLMInsightService} class uses LangChain's \texttt{ChatGoogleGenerativeAI} wrapper to call Google Gemini (specifically the \texttt{gemini\text{-}1.5\text{-}flash} model) at a low temperature of 0.2 for consistency.

The service is activated only when two environment variables are set: \texttt{ENABLE\_LLM\_INSIGHTS=true} and a valid \texttt{GOOGLE\_API\_KEY}. If either is missing, the service silently does nothing and the system falls back to the rule based output with zero degradation.

When active, the service sends a carefully crafted prompt that positions the LLM as a ``senior fleet maintenance analyst'' and provides the full prediction context as JSON, including risk level, probability, the rule based summary, the raw input data, and all the drivers. The response is parsed for a polished summary and up to three additional recommendations, which are merged with the rule based ones (prepended, deduplicated, and capped at five total).

The \texttt{insight\_source} field in the response switches from \texttt{RULES} to \texttt{GENAI\_LLM} when the LLM contributes, so the frontend can visually indicate where the insights came from. If the LLM call fails or returns something that cannot be parsed, the system falls back to rules gracefully without the user ever noticing.

\subsection{Frontend (Next.js)}

The frontend is a single page application built with Next.js~\cite{nextjs}, TypeScript, TailwindCSS, and the shadcn/ui component library.

\subsubsection{Landing Page}
The landing page is designed to make a strong first impression. It features a full screen WebGL shader background with animated gradients, a title that reveals itself character by character using a text generation effect, and a call to action button with an animated border gradient. All of these animations are implemented as reusable React components (\texttt{ShaderBackground}, \texttt{TextGenerateEffect}, \texttt{HoverBorderGradient}, \texttt{AnimatedContent}, and \texttt{FadeContent}).

\subsubsection{Prediction Page}
The prediction workspace at \texttt{/predict} is where the real work happens. It is about 727 lines of React code and offers two ways to submit data: a manual input form with five fields (mileage, engine hours, fault codes, service history, and usage patterns), or a drag and drop CSV file upload.

Both submission paths use TanStack Query mutations for state management, which handles loading indicators, error messages, and success callbacks cleanly. When a prediction comes back, the page renders a comprehensive results dashboard showing the risk level as a color coded badge, the insight source, a summary paragraph, probability and confidence numbers, data quality warnings, a batch preview table for CSV uploads, a detailed insight drivers table with direction badges and impact percentages, recommendation cards with priority labels, and a global feature importance table.

There is also a collapsible accordion section at the bottom that explains how the system works, what the risk thresholds mean, and the difference between global feature importance and row specific insight drivers.

\subsubsection{Service Layer and Types}
The frontend communicates with the backend through a TypeScript service module that provides two functions: \texttt{predictFromManualInput} for JSON submissions and \texttt{predictFromCsvFile} for multipart CSV uploads. The API base URL is automatically resolved from the \texttt{NEXT\_PUBLIC\_API\_BASE\_URL} environment variable, falling back to \texttt{localhost:8000} during development and the Vercel backend URL in production. All TypeScript type definitions mirror the backend Pydantic schemas exactly, giving us compile time safety across the full stack.

\subsection{ML Pipeline}

The trained model is saved as a single scikit learn \texttt{Pipeline} object in a pickle file. This pipeline contains two steps: a \texttt{ColumnTransformer} that applies \texttt{StandardScaler} to the 11 numerical features and \texttt{OneHotEncoder} to the 8 categorical features, followed by a tuned \texttt{RandomForestClassifier} with \texttt{max\_depth}=8, \texttt{min\_samples\_leaf}=10, \texttt{min\_samples\_split}=5, \texttt{n\_estimators}=100, and balanced class weights.

Bundling preprocessing and prediction into a single serialized object means there is no risk of preprocessing mismatch between training and serving. The notebook \texttt{model/data\_processing.ipynb} contains the complete workflow from data exploration through model training and evaluation to plot generation.

% ============================= DISCUSSION =========================
\section{Discussion}

\subsection{Model Selection}

We went with Random Forest for deployment even though Decision Tree and Gradient Boosting both hit perfect accuracy on the full feature set. The reasoning was straightforward: perfect accuracy on a synthetic dataset that contains leaking features is not a sign of a great model, it is a sign that the task is trivially solvable. We needed a model that would produce reasonable feature importance scores and generalize well if the underlying data distribution changed. Random Forest offers both of those qualities. The tuned hyperparameters (a maximum depth of 8 and a minimum leaf size of 10) further constrain the trees to prevent overfitting.

\subsection{Data Leakage Analysis}

The performance gap between Table~\ref{tab:full} and Table~\ref{tab:realistic} is enormous. On the full feature set, accuracy is at or above 96\% for every model. On the realistic set, it drops to the 70 to 80 percent range. This gap tells a clear story: the component condition features (\texttt{Brake\_Condition}, \texttt{Battery\_Status}, \texttt{Tire\_Condition}) are essentially proxies for the target variable. When the brakes are ``Worn Out,'' the vehicle always needs maintenance. That is not a prediction, it is a lookup.

In a real world deployment, these features might not even be available at prediction time. If you already know the exact condition of the brakes, you probably do not need a model to tell you the vehicle needs servicing. We kept the full feature set in the deployed pipeline because the project specification called for it, but we documented this limitation honestly throughout the report and in the codebase.

\subsection{Insight Engine Design}

Making predictions explainable was one of our top priorities from the start. We deliberately separated two concepts that are easy to confuse: global feature importance and row specific insight drivers.

Global feature importance comes from the Random Forest model itself and reflects which features the model considers most important \textit{across the entire training set}. Row specific insight drivers come from our rule engine and reflect what is happening with \textit{this particular vehicle}. The distinction matters because a feature can be globally important (like \texttt{Reported\_Issues}) while having a neutral or even protective value for a specific vehicle with zero fault codes.

We chose a deterministic rule based approach over something like SHAP or LIME for two reasons. First, it runs instantly and does not require recomputing for each prediction. Second, the rules can encode domain knowledge (like the fact that 280 days without service is a red flag) that a purely data driven explanation method would miss.

\subsection{GenAI Enhancement}

The Gemini integration was designed from the ground up to be purely additive. The rule based engine provides the foundation, and the LLM can only enhance, never override. This means the system is equally useful with or without an API key configured.

In practice, the LLM tends to produce more readable summaries and occasionally suggests actions that the rule engine does not cover, such as recommending a specific type of diagnostic tool. But since LLM outputs are not validated against any maintenance standard, we treat them as helpful suggestions rather than authoritative guidance.

\subsection{Limitations}

There are several limitations worth acknowledging. The dataset is entirely synthetic, which means the distributions, correlations, and noise patterns may not reflect what real fleet data looks like. The class imbalance was handled through balanced class weights, but techniques like SMOTE or focal loss might yield better results for the minority class. The insight engine thresholds were calibrated for this specific dataset and would need to be recalibrated for a different fleet. The LLM recommendations are not cross referenced against manufacturer service manuals. And as discussed extensively, the deployed model includes features that leak information about the target.

% ============================= DEPLOYMENT =========================
\section{Deployment}

FleetAI runs on Vercel using a two project setup that mirrors the monorepo structure.

\subsection{Backend Deployment}
The backend is deployed as a Vercel serverless function. The entrypoint file at \texttt{api/index.py} adds the \texttt{backend/} directory to \texttt{sys.path} and imports the FastAPI application. A \texttt{vercel.json} file at the repository root routes all incoming requests to this function. The trained model pickle file is included in the deployment bundle, and optional environment variables (\texttt{ENABLE\_LLM\_INSIGHTS}, \texttt{GOOGLE\_API\_KEY}, and \texttt{GENAI\_MODEL}) can be set in the Vercel dashboard to activate the Gemini integration.

\subsection{Frontend Deployment}
The frontend is deployed as a separate Vercel project with its root set to the \texttt{frontend/} directory. Vercel automatically detects the Next.js framework and handles the build process. The only required environment variable is \texttt{NEXT\_PUBLIC\_API\_BASE\_URL}, which should point to the deployed backend domain.

\subsection{Verification}
After each deployment we verify that the health endpoint returns \texttt{\{``status'': ``ok''\}}, that the auto generated API documentation is accessible at \texttt{/docs}, and that a test prediction with sample JSON and CSV payloads returns the expected response structure.

\subsection{POST /predict (CSV)}
For batch predictions, send a multipart form with a \texttt{file} field containing a CSV. Required columns are \texttt{mileage}, \texttt{engine\_hours}, \texttt{fault\_codes}, \texttt{service\_history}, and \texttt{usage\_patterns}. The response includes a \texttt{predictions[]} array and a top level summary of the highest risk row.

\subsection{Risk Thresholds}
\begin{itemize}
    \item \texttt{LOW}: probability below 0.30
    \item \texttt{MEDIUM}: probability from 0.30 up to 0.70
    \item \texttt{HIGH}: probability at or above 0.70
\end{itemize}

% ============================= CONCLUSION =========================
\section{Conclusion}

We built FleetAI, a vehicle maintenance prediction system that goes beyond just outputting a risk score. Every prediction comes with structured insight drivers, prioritized recommendations, and data quality warnings. An optional Gemini LLM layer adds polished natural language summaries when configured, while the rule based engine ensures the system is always useful even without any external API.

Our analysis of the Kaggle dataset uncovered significant data leakage through component condition features. We reported this transparently and showed results on both the full and clean feature sets. The tuned Random Forest classifier achieves 94.07\% accuracy on the test set and is deployed as a REST API supporting both JSON and CSV inputs, with a modern Next.js dashboard providing an intuitive interface for fleet managers.

Looking ahead, we plan to explore several directions:

\begin{enumerate}
    \item Retrieval augmented generation using maintenance manuals stored in Pinecone, so the LLM can ground its recommendations in manufacturer documentation.
    \item Agentic workflow orchestration using LangGraph for multi step analysis tasks.
    \item Persistent vehicle health tracking through PostgreSQL using Piccolo ORM, for which table definitions already exist in the codebase.
    \item Retraining with real world fleet data and techniques like SMOTE for improved minority class performance.
    \item A mobile application for field technicians who need to check vehicle status on the go.
\end{enumerate}
